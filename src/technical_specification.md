# Техническое задание (стандарт IEEE 29148-2011)*
**Разработка информационной системы для самостоятельного обслуживания техники с использованием ИИ помощника**

**Версия:** 1.0  
**Дата:** 26 ноября 2025 г.

---

## 1. Введение

### 1.1 Назначение
Назначением данного документа является полное и однозначное определение требований к программному обеспечению "Информационной системы самостоятельного обслуживания электроники с ИИ-помощником". Система предназначена для диагностики и ремонта бытовой электроники (мониторы, микроволновые печи, телевизоры, аудиоколонки). Документ служит основой для согласования между Заказчиком и Разработчиком, а также для проектирования, реализации, тестирования и приемки Системы.

### 1.2 Область действия
Система представляет собой кроссплатформенное решение (веб- и мобильное приложение), которое позволяет пользователям самостоятельно диагностировать и устранять неисправности бытовой электроники.

**Ключевые функции включают:**
*   Интеллектуальный диалоговый помощник.
*   Интерактивные инструкции.
*   Интеграцию с каталогом электронных компонентов и сервисными центрами.

## 2. Общее описание

### 2.1 Взаимодействие с продуктом
Система работает как автономное приложение, взаимодействуя с внешними сервисами:

*   **Платежные шлюзы** (для оплаты компонентов).
*   **API систем снабжения и каталогов** электронных компонентов (Поиск, сравнение и заказ компонентов).
*   **Платформы для размещения пользовательского контента** (форум, база знаний).
*   **API-базы данных электронных компонентов** (например, Octopart, LCSC) для актуальных данных о наличии и аналогах.
*   **Сервисы нотификации** (SMS, E-mail).

### 2.2 Функции продукта
1.  **Проведение диагностики неисправности:** Единый сквозной процесс, при котором ИИ-помощник на основе описания симптомов пользователем проводит первичный анализ и, при необходимости, верифицирует диагноз через пошаговую проверку компонентов с помощью инструментов (мультиметр).
2.  **Оценка рисков и сложности ремонта:** Автоматическое определение уровня опасности (риск поражения током, повреждения устройства) и технической сложности операций (пайка, настройка) с выдачей рекомендации о целесообразности самостоятельного ремонта.
3.  **Формирование персонализированного плана ремонта:** Генерация пошагового алгоритма действий и перечня необходимого инструментария, адаптированного под навыки пользователя (начинающий/опытный).
4.  **Предоставление экспертного решения:** Обеспечение доступа к базе знаний сообщества, обсуждениям и платным консультациям инженеров-экспертов для обработки сложных кейсов.
5.  **Ведение электронного журнала ремонтов и формирование отчетности:** Систематический учет истории обслуживания каждого устройства с возможностью генерации отчетных документов.
6.  **Интеграция с системами снабжения и каталогами компонентов:** Поиск, сравнение и заказ необходимых электронных компонентов и инструментов через подключение к внешним каталогам и маркетплейсам.
7.  **Управление структурой знаний и онтологией техники:** Административный инструмент для наполнения и обновления базы знаний, моделей устройств, симптоматики и решений, обеспечивающий развитие и актуальность системы.

### 2.3 Классы пользователей и характеристики

| Класс пользователя | Характеристики |
| :--- | :--- |
| **Гость** | Может провести базовую диагностику. Доступ ограничен. |
| **Пользователь** | Имеет базовые навыки (желание работать с отверткой, паяльником). Имеет полный доступ к функциям. |
| **Администратор** | Управляет пользователями, контентом БЗ, моделями ИИ. |
| **Эксперт** | Подтвержденный специалист. Получает сложные кейсы, может править и дополнять БЗ. |

### 2.4 Операционное окружение
*   **Аппаратное:** Серверы x86-64. Мобильные устройства с iOS 14+/Android 8.0+ с камерой.
*   **Программное:** ОС: Windows 7,8,10,11, Ubuntu 20.04 LTS; СУБД: PostgreSQL; Бэкенд: Python (ML), Node.js; Фронтенд: React/React Native.
*   **Безопасность:** Обязательное использование TLS 1.2+ для всех соединений.

### 2.5 Ограничения проектирования и реализации
*   Система должна включать обязательные предупреждения о безопасности при работе с высоковольтными компонентами (например, в телевизорах и микроволновках).
*   ИИ-помощник должен быть обучен на распознавание технических терминов, связанных с электроникой (например, "инвертор", "динамик", "предохранитель", "конденсатор").
*   Интерфейс должен быть локализован на русский язык.

## 3. Требования к системе

### 3.1 Функциональные требования (ФТ)

#### 3.1.1 Модуль диагностики неисправностей
*   **ФТ-001:** Система должна предоставлять интерфейс чата для текстового или голосового взаимодействия с ИИ-помощником.
*   **ФТ-002:** ИИ-помощник должен определять тип и модель устройства на основе введенных пользователем данных (серийный номер, штрих-код, модель) или через уточняющий диалог.
*   **ФТ-003:** ИИ-помощник должен проводить пользователя по сценарию диагностики, задавая уточняющие вопросы о симптомах неисправности ("не включается", "нет изображения", "посторонний шум").
*   **ФТ-004:** На основе анализа симптомов ИИ-помощник должен генерировать гипотезу о наиболее вероятной причине неисправности (например, "перегорел предохранитель F1", "неисправность блока питания").
*   **ФТ-005:** Для верификации гипотезы Система должна предоставлять пользователю пошаговые инструкции по проверке компонентов (например, "прозвоните предохранитель", "измерьте напряжение на конденсаторе C105") с использованием инструментов (мультиметр).

#### 3.1.2 Модуль оценки рисков и планирования ремонта
*   **ФТ-010:** На основе результата диагностики Система должна автоматически оценивать и присваивать уровни:
    *   а) **Риска:** Безопасность проведения ремонта для пользователя и устройства (низкий, средний, высокий).
    *   б) **Сложности:** Технической сложности требуемых операций (например, "замена компонента пайкой", "программная настройка").
*   **ФТ-011:** Система должна выдавать итоговую рекомендацию на основе оценок риска и сложности (например, "Ремонт возможен самостоятельно", "Рекомендуется обратиться в сервисный центр").
*   **ФТ-012:** Система должна формировать персонализированный план ремонта, включающий:
    *   а) Пошаговый алгоритм действий.
    *   б) Перечень необходимых инструментов и материалов.
    *   в) Адаптацию сложности инструкций и состава инструментов в зависимости от выбранного пользователем уровня навыков ("Новичок" / "Опытный").

#### 3.1.3 Модуль интеграции с системами снабжения
*   **ФТ-020:** Система должна по идентифицированному в ходе диагностики компоненту (например, "конденсатор 1000мкФ 16В") находить его в подключенных каталогах, отображая актуальные данные о наличии, цене и технических аналогах.
*   **ФТ-021:** Система должна предоставлять пользователю возможность оформлять заказ на найденные компоненты и инструменты через интегрированную корзину и подключенный платежный шлюз.

#### 3.1.4 Модуль сообщества и базы знаний
*   **ФТ-030:** Система должна предоставлять доступ к форуму (сообществу), где пользователи (как обычные, так и эксперты-ремонтники) могут:
    *   а) Публиковать вопросы и ответы.
    *   б) Обсуждать проблемы и решения.
    *   в) Искать информацию по схожим неисправностям.
*   **ФТ-031:** Система должна предоставлять доступ к структурированной Базе Знаний, содержащей статьи, мануалы и типовые решения по ремонту.
*   **ФТ-032:** Система должна позволять пользователю напрямую переносить данные текущей сессии диагностики (историю диалога, выявленные симптомы) в новый пост на форуме для получения помощи.

#### 3.1.5 Модуль журналирования и отчетности
*   **ФТ-040:** Система должна автоматически вести электронный журнал для каждого устройства пользователя, сохраняя историю всех проведенных диагностик, их результатов и выполненных ремонтов.
*   **ФТ-041:** Система должна предоставлять возможность формирования и экспорта отчета о завершенном ремонте в формате PDF, включающего диагноз, использованные компоненты и выполненные действия.

#### 3.1.6 Модуль управления знаниями и администрирования
*   **ФТ-050:** Администратор должен иметь возможность управлять онтологией техники и структурой знаний: добавлять, редактировать и удалять модели устройств, их компоненты, типовые симптомы неисправностей и решения.
*   **ФТ-051:** Администратор должен иметь возможность модерировать контент форума и Базы Знаний.
*   **ФТ-052:** При переходе пользователя к инструкциям, связанным с работой с высоким напряжением, Система должна выводить обязательное предупреждение о риске с требованием подтверждения осведомленности.

### 3.2 Требования к внешним интерфейсам
*   **ТВИ-001:** Пользовательский интерфейс должен быть адаптирован для разрешений от 320px до 1920px по ширине и от 500px до 1080px по длине.
*   **ТВИ-002:** API для интеграции с CRM и каталогами компонентов должен использовать REST, JSON и аутентификацию по OAuth 2.0.

### 3.3 Требования к производительности
*   **ТП-001:** Время отклика ИИ-помощника на сообщение ≤ 3 секунд.
*   **ТП-002:** Система должна обрабатывать до 5000 одновременных сессий диагностики.

### 3.4 Специальные требования
*   **ТС-001:** Модель ИИ должна достигать точности первичной диагностики не менее 80% для топ-1000 самых популярных моделей электроники.

### 3.5 Атрибуты качества
*   **Надежность:** Доступность Системы ≥ 99%.
*   **Удобство использования:** Индекс удобства использования (System Usability Scale) ≥ 75 баллов по результатам тестирования с целевой аудиторией.
*   **Безопасность:** Все пароли должны храниться в хешированном виде. Доступ к платежным данным осуществляется только через токенизацию.
*   **Сопровождаемость:** Система должна предоставлять детальные логи всех операций ИИ для его последующего дообучения и анализа ошибок.

## 4. Нефункциональные требования

### 4.1 Требования к производительности
*   **ТП-001:** Время отклика системы (время между действием пользователя и ответом системы) для любых операций в интерфейсе не должно превышать 2 секунды при стандартной нагрузке.
*   **ТП-002:** Время отклика ИИ-помощника на текстовое сообщение пользователя не должно превышать 3 секунды.
*   **ТП-003:** Система должна выдерживать пиковую нагрузку до 5 000 одновременных активных сессий пользователей.
*   **ТП-004:** Время обработки транзакции (оформление заказа на компонент) не должно превышать 4 секунд.

### 4.2 Требования к безопасности
*   **ТБ-001:** **Аутентификация:** Все пользователи должны проходить аутентификацию для доступа к персональным данным и функциям ремонта. Минимальные требования к паролю: 8 символов, заглавные и строчные буквы, цифры.
*   **ТБ-002:** **Авторизация:** Доступ к данным и функционалу должен строго контролироваться на основе ролей пользователей (Гость, Пользователь, Администратор).
*   **ТБ-003:** **Защита данных:** Все персональные данные (ФИО, контакты, история ремонтов) и платежная информация должны передаваться по протоколу TLS 1.2+ и храниться в зашифрованном виде.
*   **ТБ-004:** **Конфиденциальность:** Система не должна предоставлять данным одной учетной записи другим пользователям. История диагностики и ремонтов доступна только владельцу учетной записи.
*   **ТБ-005:** **Аудит:** Все критичные действия (вход в систему, изменение данных в БЗ администратором, эскалация заявки, финансовые операции) должны логироваться с указанием пользователя, времени и сути действия. Срок хранения логов - не менее 1 года.
*   **ТБ-006:** **Предупреждения:** Система должна выдавать обязательные для подтверждения предупреждения о рисках поражения электрическим током при работе с высоковольтными узлами (блоки питания телевизоров, микроволновых печей).

### 4.3 Требования к атрибутам качества программного обеспечения
*   **ТК-001: Надежность:**
    *   Доступность (uptime) системы должна составлять не менее 99% в расчете на месяц.
    *   Среднее время наработки на отказ (MTBF) должно составлять не менее 720 часов.
*   **ТК-002: Удобство использования:**
    *   Интерфейс должен быть интуитивно понятен для пользователей с техническим складом ума, но без глубоких познаний в электронике.
    *   Оценка System Usability Scale (SUS) после приемочных испытаний должна быть не менее 75 баллов.
*   **ТК-003: Сопровождаемость:**
    *   Система должна быть построена по модульному принципу.
    *   Исходный код должен сопровождаться документацией, описывающей его структуру и принципы работы ключевых модулей (ИИ).
    *   Все публичные API-методы должны быть описаны с использованием OpenAPI Specification.
*   **ТК-004: Переносимость:**
    *   Веб-интерфейс должен корректно работать в последних версиях браузеров Chrome, Firefox, Safari, Opera, Microsoft Edge.
    *   Мобильное приложение должно быть совместимо с iOS 14+ и Android 10+.

## 5. Внешние интерфейсы

### 5.1 Пользовательские интерфейсы
*   **ТВИ-001:** Веб-интерфейс должен быть адаптивным и корректно отображаться на экранах с разрешением от 320px (мобильные устройства) до 3840px (4K мониторы).
*   **ТВИ-002:** Мобильное приложение должно иметь интерфейс, оптимизированный для управления одной рукой, с учетом размеров современных смартфонов.

### 5.2 Программные интерфейсы
*   **ТПИ-001: Интерфейс с платежным шлюзом:** Система должна взаимодействовать с внешним платежным шлюзом (например, Stripe, CloudPayments) через REST API для проведения безопасных финансовых транзакций. Формат данных - JSON.
*   **ТПИ-002: Интерфейс с системами снабжения:** Система должна интегрироваться с каталогами электронных компонентов и инструментов (например, через API LCSC, Octopart) через REST API для получения актуальных данных о наличии, ценах, технических характеристиках и аналогах компонентов.
*   **ТПИ-003: Интерфейс с сервисами модерации:** Система должна интегрироваться с внешними сервисами модерации контента через REST API для автоматической проверки пользовательского контента, публикуемого на форуме и в базе знаний.

### 5.3 Интерфейсы связи
*   **ТИС-001:** Все внешние коммуникации должны использовать защищенные протоколы: HTTPS для веб-трафика и TLS для прочих данных.
*   **ТИС-002:** Система должна поддерживать отправку уведомлений пользователям по протоколам SMTP (для E-mail) и SMS (через интеграцию с SMS-шлюзом).

## 6. Другие требования

### 6.1 Требования к документации
*   **ТД-001: Руководство пользователя:** Должно быть предоставлено интерактивное руководство, интегрированное в приложение, описывающее основные сценарии работы: регистрация, проведение диагностики, использование персонализированных планов ремонта и работа с форумом.
*   **ТД-002: Техническая документация:** Должна быть разработана документация для администраторов системы, включающая руководство по развертыванию, администрированию базы знаний и онтологии техники, и мониторингу работы ИИ-моделей.
*   **ТД-003: Руководство по API:** Должна быть предоставлена публичная документация по всем внешним API, сгенерированная на основе OpenAPI Specification.

### 6.2 Требования к лицензированию
*   **ТЛ-001:** Используемые сторонние библиотеки, SDK и шрифты должны иметь лицензии, совместимые с коммерческим использованием продукта.
*   **ТЛ-002:** Права на исходный код и объектный код системы принадлежат Заказчику.

### 6.3 Правовые требования и требования соответствия
*   **ТС-001:** Система должна соответствовать Федеральному закону № 152-ФЗ "О персональных данных".
*   **ТС-002:** Все инструкции по ремонту должны быть классифицированы в соответствии с автоматически присваиваемой системой категорией риска. Доступ к инструкциям с высоким уровнем риска должен быть ограничен обязательным для подтверждения предупреждением о необходимости специальных навыков и соблюдении правил электробезопасности.
*   **ТС-003:** В случае оформления заказа на компоненты, система должна обеспечивать формирование электронного чека в соответствии с законодательством РФ.
*   **ТС-004:** Система должна обеспечивать соблюдение законодательства о распространении информации (ФЗ-149) путем проверки пользовательского контента на форуме и в базе знаний на соответствие установленным требованиям.

### 6.4 Требования к тестированию
*   **ТТ-001:** Должны быть разработаны тестовые сценарии, покрывающие все основные и альтернативные потоки использования, описанные в Приложении А.
*   **ТТ-002:** Должно быть проведено нагрузочное тестирование для проверки выполнения требований ТП-003 (5000 одновременных сессий).
*   **ТТ-003:** Должно быть проведено тестирование безопасности (пентест) для выявления уязвимостей.
*   **ТТ-004:** Должно быть проведено тестирование точности формирования персонализированных планов ремонта для различных категорий пользователей (начинающий/опытный).

### 6.5 Требования к хранению данных
*   **ТХД-001:** Резервное копирование базы данных и пользовательских файлов должно производиться ежедневно.
*   **ТХД-002:** Срок хранения персональных данных пользователя определяется Политикой конфиденциальности, но не может быть менее 1 года с момента последней активности.
*   **ТХД-003:** Логи безопасности должны храниться в течение 3 лет.
*   **ТХД-004:** Электронные журналы ремонтов и отчеты пользователей должны храниться в течение 3 лет с возможностью экспорта пользователем.
